Streams

The first thing I want to talk about is streams.

record Stream (A : Type) : Type where
  coinductive
  field head : A
        tail : Stream A

This representation is coinductive: the type above contains infinite values. Agda, unlike Haskell, treats inductive and coinductive types differently (this is why we need the coinductive keyword in the definition). One of the differences is that it doesn‚Äôt check termination for construction of these values:

alternating : Stream Bool
alternating .head       = true
alternating .tail .head = false
alternating .tail .tail = alternating

alternating :: [Bool]
alternating = True : False : alternating

We have the equivalent in Haskell on the right. We‚Äôre also using some fancy syntax for the Agda code: copatterns (Abel and Pientka 2013).

Note that this type is only definable in a language with some notion of laziness. If we tried to define a value like alternating above in OCaml we would loop. Haskell has no problem, and Agda‚Äîthrough its coinduction mechanism‚Äîcan handle it as well.

There is, however, an isomorphic type that can be defined without coinduction:

‚Ñï-Stream : Type ‚Üí Type
‚Ñï-Stream A = ‚Ñï ‚Üí A

‚Ñï-alternating : ‚Ñï-Stream Bool
‚Ñï-alternating zero          = true
‚Ñï-alternating (suc zero)    = false
‚Ñï-alternating (suc (suc n)) = ‚Ñï-alternating n

(notice that, in this form, the function ‚Ñï-alternating is the same function as even : ‚Ñï ‚Üí Bool)

In fact, we can convert from the coinductive representation to the inductive one. This conversion function is more familiarly recognisable as the indexing function:

_[_] : Stream A ‚Üí ‚Ñï-Stream A
xs [ zero  ] = xs .head
xs [ suc n ] = xs .tail [ n ]

I‚Äôm not just handwaving when I say the two representations are isomorphic: we can prove this isomorphism, and, in Cubical Agda, we can use this to transport programs on one representation to the other.

Proof of isomorphism

tabulate xs .head = xs zero
tabulate xs .tail = tabulate (xs ‚àò suc)

stream-rinv : (xs : Stream A) ‚Üí tabulate (xs [_]) ‚â° xs
stream-rinv xs i .head = xs .head
stream-rinv xs i .tail = stream-rinv (xs .tail) i

stream-linv xs (suc n) = stream-linv (xs ‚àò suc) n

stream-reps : ‚Ñï-Stream A ‚áî Stream A
stream-reps .fun = tabulate
stream-reps .inv = _[_]
stream-reps .rightInv = stream-rinv
stream-reps .leftInv xs = funExt (stream-linv xs)

One final observation about streams: another way to define a stream is as the cofree comonad of the identity functor.

record Cofree (F : Type ‚Üí¬†Type) (A : Type) : Type where
  coinductive
  field root : A
        step : F (Cofree F A)

ùíû-Stream : Type ‚Üí Type
ùíû-Stream = Cofree id

Concretely, the Cofree F A type is a possibly infinite tree, with branches shaped like F, and internal nodes labelled with A. It has the following characteristic function:

{-# NON_TERMINATING #-}

trace œï œÅ x .step = map (trace œï œÅ) (œÅ x)

Like how the free monad turns any functor into a monad, the cofree comonad turns any functor into a comonad. Comonads are less popular and widely-used than monads, as there are less well-known examples of them. I have found it helpful to think about comonads through spatial analogies. A lot of comonads can represent a kind of walk through some space: the extract operation tells you ‚Äúwhat is immediately here‚Äù, and the duplicate operation tells you ‚Äúwhat can I see from each point‚Äù. For the stream, these two operations are inhabited by head and the following:

duplicate : Stream A ‚Üí Stream (Stream A)
duplicate xs .head = xs
duplicate xs .tail = duplicate (xs .tail)

Generalising Streams

There were three key observations in the last section:

  1. Streams are coinductive. This requires a different termination checker in Agda, and a different evaluation model in strict languages.
  2. They have an isomorphic representation based on indexing. This isomorphic representation doesn‚Äôt need coinduction or laziness.
  3. They are a special case of the cofree comonad.

Going forward, we‚Äôre going to look at generalisations of streams, and we‚Äôre going to see what these observations mean in the contexts of the new generalisations.

The thing we‚Äôll be generalising is the index of the stream. Currently, streams are basically structures that assign a value to every ‚Ñï: what does a stream of‚Äîfor instance‚Äîrational numbers look like? To drive the intuition for this generalisation let‚Äôs first look at the comonad instance on the ‚Ñï-Stream type:

‚Ñï-extract : ‚Ñï-Stream A ‚Üí A
‚Ñï-extract xs = xs zero

‚Ñï-duplicate : ‚Ñï-Stream A ‚Üí ‚Ñï-Stream (‚Ñï-Stream A)
‚Ñï-duplicate xs zero    = xs
‚Ñï-duplicate xs (suc n) = ‚Ñï-duplicate (xs ‚àò suc) n

This is the same instance as is on the Stream type, transported along the isomorphism between the two types (we could have transported the instance automatically, using subst or transport; I have written it out here manually in full for illustration purposes).

The ‚Ñï-duplicate method here can changed a little to reveal something interesting:

‚Ñï-duplicate‚ÇÇ : ‚Ñï-Stream A ‚Üí ‚Ñï-Stream (‚Ñï-Stream A)
‚Ñï-duplicate‚ÇÇ xs zero    m = xs m
‚Ñï-duplicate‚ÇÇ xs (suc n) m = ‚Ñï-duplicate‚ÇÇ (xs ‚àò suc) n m

‚Ñï-duplicate‚ÇÉ : ‚Ñï-Stream A ‚Üí ‚Ñï-Stream (‚Ñï-Stream A)
‚Ñï-duplicate‚ÇÉ xs n m = xs (go n m)
  where
  go : ‚Ñï ‚Üí ‚Ñï ‚Üí ‚Ñï
  go zero    m = m
  go (suc n) m = suc (go n m)

‚Ñï-duplicate‚ÇÑ : ‚Ñï-Stream A ‚Üí ‚Ñï-Stream (‚Ñï-Stream A)
‚Ñï-duplicate‚ÇÑ xs n m = xs (n + m)

In other words, duplicate basically adds indices.

There is something distinctly monoidal about what‚Äôs going on here: taking the (‚Ñï, +, 0) monoid as focus, the extract method above corresponds to the monoidal empty element, and the duplicate method corresponds to the binary operator on monoids. In actual fact, there is a comonad for any function from a monoid, often called the Traced comonad.

Traced : Type ‚Üí Type ‚Üí Type

extract·µÄ : ‚¶É _ : Monoid E ‚¶Ñ ‚Üí Traced E A ‚Üí A

duplicate·µÄ : ‚¶É _ : Monoid E ‚¶Ñ ‚Üí Traced E A ‚Üí Traced E (Traced E A)
duplicate·µÄ xs e‚ÇÅ e‚ÇÇ = xs (e‚ÇÅ ‚àô e‚ÇÇ)

Reifying Traced

The second observation we made about streams was that they had an isomorphic representation which didn‚Äôt need coinduction. What we can see above, with Traced, is a representation that also doesn‚Äôt need coinduction. So what is the corresponding coinductive representation? What does a generalised reified stream look like?

So the first approach to reifying a function to a data structure is to simply represent the function as a list of pairs.

C-Traced : Type ‚Üí Type ‚Üí Type
C-Traced E A = Stream (E √ó A)

This representation obviously isn‚Äôt ideal: it isn‚Äôt possible to construct an isomorphism between C-Traced and Traced. We can‚Äîkind of‚Äîgo in one direction, but even that function isn‚Äôt terminating:

{-# NON_TERMINATING #-}
lookup-env : ‚¶É _ : IsDiscrete E ‚¶Ñ ‚Üí C-Traced E A ‚Üí Traced E A
lookup-env xs x = if does (x ‚âü xs .head .fst)
                     then xs .head .snd
                     else lookup-env (xs .tail) x

I‚Äôm not too concerned with being fast and loose with termination and isomorphisms for the time being, though. At the moment, I‚Äôm just interested in exploring the relationship between streams and the indexing functions.

As a result, let‚Äôs try and push on this representation a little and see if it‚Äôs possible to get something interesting and almost isomorphic.

Segmented Streams

To get a slightly nicer representation we can exploit the monoid a little bit. We can do this by storing offsets instead of the absolute indices for each entry. The data structure I have in mind here looks a little like this:

Above is a stream containing the values x, y, and z. Instead of each value corresponding to a single entry in the stream, however, they each correspond to a segment. The value x, for instance, labels the first segment in the stream, which has a length given by a. y labels the second segment, with length b, z with length c, and so on.

The Traced version of the above structure might be something like this:

str :: Traced m a
str i | i < a         = x
      | i < a + b     = y
      | i < a + b + c = z
      | ...

So the index-value mapping is also segmented. The stream, in this way, is kind of like a ruler, where different values mark out different quantities along the ruler, and the index function takes in a quantity and tells you which entry in the ruler that quantity corresponds to.

In code, we might represent the above data structure with the following type:

record Segments (E : Type) (A : Type) : Type where
  field
    length : E
    label  : A
    next   : Segments E A

open Segments

The question is, then, how do we convert this structure to an Traced representation?

Monuses

We need some extra operations on the monoid in the segments in order to enable this conversion to the Traced representation. The extra operations are encapsulated by the monus algebra: I wrote about this in the paper I submitted with Nicolas Wu to ICFP last year (2021). It‚Äôs a simple algebra on monoids which basically encapsulates monoids which are ordered in a sensible way.

The basic idea is that we construct an order on monoids which says ‚Äúx is smaller than y if there is some z that we can add to x to get to y‚Äù.

_‚âº_ : ‚¶É _ : Monoid A ‚¶Ñ ‚Üí A ‚Üí A ‚Üí Type _
x ‚âº y = ‚àÉ z √ó (y ‚â° x ‚àô z)

A monus is a monoid where we can extract that z, when it exists. On the monoid (‚Ñï, +, 0), for instance, this order corresponds to the normal ordering on ‚Ñï.

Extracting the z above corresponds to a kind of difference operator:

_‚à∏_ : ‚Ñï ‚Üí ‚Ñï ‚Üí ‚Ñï
x     ‚à∏ zero  = x
suc x ‚à∏ suc y = x ‚à∏ y
_     ‚à∏ _     = zero

This operator is sometimes called the monus. It is a kind of partial, or truncating, subtraction:

_ : 5 ‚à∏ 2 ‚â° 3
_ = refl

_ : 2 ‚à∏ 5 ‚â° 0
_ = refl

And, indeed, this operator ‚Äúextracts‚Äù the z, when it exists.

‚à∏‚Äøis-monus : ‚àÄ x y ‚Üí (x‚âºy : x ‚âº y) ‚Üí y ‚à∏ x ‚â° fst x‚âºy
‚à∏‚Äøis-monus zero    _       (z , y‚â°0+z) = y‚â°0+z
‚à∏‚Äøis-monus (suc x) (suc y) (z , y‚â°x+z) = ‚à∏‚Äøis-monus x y (z , suc-inj y‚â°x+z)
‚à∏‚Äøis-monus (suc x) zero    (z , 0‚â°x+z) = ‚ä•-elim (zero‚â¢suc 0‚â°x+z)

Our definition of a monus is simple: a monus is anything where the order ‚âº, sometimes called the ‚Äúalgebraic preorder‚Äù, is total and antisymmetric. This is precisely what lets us write a function which takes the Segments type and converts it back to the Traced type.

{-# NON_TERMINATING #-}

... | yes (j , i‚â°xs‚Çó‚àôj) = Segments‚ÜíTraced (xs .next) j
... | no  _             = xs .label

This function takes an index, and checks if that length is greater than or equal to the first segment in the stream of segments. If it is, then it continues searching through the rest of the segments with the index reduced by the size of that first segment. If not, then it returns the label of the first segment.

Taking the old example, we are basically converting to ‚à∏ from +:

str :: Traced m a
str i | i         < a = x
      | i ‚à∏ a     < b = y
      | i ‚à∏ a ‚à∏ b < c = z
      | ...

The first issue here is that this definition is not terminating. That might seem an insurmountable problem at first‚Äîwe are searching through an infinite stream, after all‚Äîbut notice that there is one paremeter which is decreasing on each recursive call: the index. Well, it only decreases if the segment is non-zero: this can be enforced by changing the definition of the segments type:

Trying to build an isomorphism

So the ‚Ñ±-Segments type is interesting, but it only really gives one side of the isomorphism. There is no way to write a function Traced E A ‚Üí ‚Ñ±-Segments E A.

The problem is that there‚Äôs no way to get the ‚Äúnext‚Äù segment from a function E ‚Üí A. We can find the label of the first segment, by applying the function to Œµ, but there‚Äôs no real way to figure out the size of this segment. We can change Traced little to provide this size, though.

State Comonad

There is a comonad on state (Waern 2018; Kmett 2018) that is different from store. Notice that above the Ind type has the same type (almost) as State E A.

This is interesting in two ways: first, it gives some concrete, spatial intuition for what‚Äôs going on with the state comonad.

Second, it gives a kind of interesting monad instance on the stream. If we apply the Ind‚Üí‚Ñ±-Segments function to the implementation of join on state, we should get a join on ‚Ñ±-Segments. And we do!

First, we need to redefine Ind to the following:

Again, we‚Äôre focusing on the ys section here, which occupies the segment from a to a ‚àô b. After join, this segment is occupied by three elements, x ∏, y ∏, and z ∏.

Notice that this isn‚Äôt quite the normal join on streams. That join takes a stream of streams, and turns the ith entry into the ith entry in the underlying stream. It‚Äôs a diagonalisation, in other words.

This one is kind of similar, but it takes chunks of the outer stream.

Theory

All of this so far is very hand-wavy. We have an almost isomorphism (a split surjection, to be precise), but not much in the way of concrete theoretical insights, just some vague gesturing towards spatial metaphors and so on.

Thankfully, there are two seperate areas of more serious research that seem related to the stuff I‚Äôve talked about here. The first is update monads and directed containers, and the second is graded comonads. I think I understand graded comonads and the related work better out of the two, but update monads and directed containers seems more closely related to what I‚Äôm doing here.

Update Monads and Directed Containers

There are a few papers on this topic: Ahman, Chapman, and Uustalu (2012), Ahman and Uustalu (2013; Ahman and Uustalu 2014; Ahman and Uustalu 2016).

The first of these, ‚ÄúWhen Is a Container a Comonad?‚Äù constructs, as the title suggests, a class for containers which are comonads in a standard way.

Here‚Äôs the definition of a container:

Containers are a generic way to describe a class of well-behaved functors. Any container is a pair of a shape and position. Lists, for instance, are containers, where their shape is described by the natural numbers (the shape here is the length of the list). The positions in such a list are the numbers smaller than the length, in dependently-typed programming we usually use the Fin type for this:

The benefit of using containers is that it gives a standard, generic, and composable way to construct functors that have some nice properties (like strict positivity). They‚Äôre pretty annoying to use in practice, though, which is a shame.

Directed containers are container that have three extra operations.

  • A tail-like operation, where a position can be converted into the shape of containers that the suffic from that position.
  • A head-like operation, where you can always return the root position.
  • A +-like operation, where you take a position on some tail and translate it into a position on the original container, by adding it.

As the paper observes, these are very similar to a ‚Äúdependently-typed‚Äù version of the monoid methods. This seems to me to be very similar to the indexing stuff we were doing earlier on.

The real interesting part is in the paper ‚ÄúUpdated Monads: Cointerpreting Directed Containers‚Äù (Ahman and Uustalu 2014). This paper presents a variant on state monads, called ‚Äúupdate monads‚Äù.

I‚Äôm still in the early stages of understanding all of this material, but at the moment graded comonads and transformers are concepts that I‚Äôm much more familiar and comfortable with.

The idea behind graded monads and comonads is similar to the idea behind any indexed monad: we‚Äôre adding an extra type parameter to the monad or type, which can constrain the operations involved. The graded monads and comonads use a monoid as that index. This works particularly nicely, in my opinion: just allowing any index at all sometimes feels a little unstructured. Conclusion

Comonads are much less widely used than monads in Haskell and similar languages. Part of the reason, I think, is that they‚Äôre too powerful in a non-linear language. Monads are often used to model sublanguages where it‚Äôs possible to introduce ‚Äúspecial‚Äù variables which interact with the monadic context.

pyth = do
  x <- [1..10]
  y <- [1..10]
  z <- [1..10]
  guard (x*x + y*y == z*z)
  return (x,y,z)

The x variable here semantically spans over the range [1..10]. In the following two examples we see the semantics of state and maybe:

sum :: [Int] -> Int
sum xs = flip evalState 0 $ do
  put 0
  for_ xs $ \x -> do
    n <- get
    put (n + x)
  m <- get
  return m
data E = Lit Int | E :+: E | E :/: E

eval :: E -> Maybe Int
eval (Lit n) = n
eval (xs :+: ys) = do x <- eval xs
                      y <- eval ys
                      return (x + y)
eval (xs :/: ys) = do x <- eval xs
                      y <- eval ys
                      guard (y /= 0)
                      return (x / y)

The variables n and m introduced in the state example are ‚Äúspecial‚Äù because their values depend on the computations that came before. In the maybe example the variables introduced could be Nothing.

You can‚Äôt do the same thing with comonads because you‚Äôre always able to extract the ‚Äúspecial‚Äù variable with extract :: m a -> a. Instead of having special variable introduction, comonads let you have special variable elimination. But, since Haskell isn‚Äôt linear, you can always just discard a variable so this isn‚Äôt much use.

Looking at the maybe example, we have a function eval :: E -> Maybe Int that introduces an Int variable with a ‚Äúcatch‚Äù: it is wrapped in a Maybe. We want to use the eval function as if it were a normal function E -> Int, with all of the bookkeeping managed for us: that‚Äôs what monads and do notation (kind of) allow us to do.

An analagous example with comonads might be having a function consume :: m V -> String. This ‚Äúhandles‚Äù a V value, but the ‚Äúcatch‚Äù is that it needs an m context to do so. If we want to treat the consume function as if it were a normal function V -> String then comonads (and codo notation Orchard and Mycroft 2013) would be a perfect fit.

The reason that this analagous case doesn‚Äôt arise very often is that we don‚Äôt have many handlers that look like m V -> String in Haskell. Why? Because if we want to ‚Äúhandle‚Äù a V we can just discard it: as a non-linear language, you do not need to perform any ceremony to discard a variable in Haskell.

Graded comonads, though, seem to be much more useful than normal comonads. I think it is becuase they basically get rid of the m a -> a function, changing it into a much more restricted form. In this way, they give a kind of small linear language, but just for the monoidal type parameter.

And there are a lot of uses for the graded comonads. Above we‚Äôve used them for termination checking. A recursive function might have the form a -> b, where a is the thing being recursed on. If we‚Äôre using well-founded recursion to show that it‚Äôs terminating, though, we add an extra parameter, an Acc _<_ proof, turning this function into Acc _<_ w √ó a -> b. The Acc _<_ here is the graded comonad, and this recursive function is precisely the ‚Äúhandler‚Äù.

Other examples might be provacy or permissions: a function might be able to work on some value, but only if it has particular permission regarding that value. The permission here is the monoid.

There are other examples I‚Äôm sure, those are just the couple that I have been thinking about.
